    Introduction

    - Remain polite, courteous, respectful and constructive
    throughout the evaluation process. The well-being of the community
    depends on it.

    - Identify with the person (or the group) evaluated the eventual
    dysfunctions of the work. Take the time to discuss
    and debate the problems you have identified.

    - You must consider that there might be some difference in how your
    peers might have understood the project's instructions and the
    scope of its functionalities. Always keep an open mind and grade
    him/her as honestly as possible. The pedagogy is valid only and
    only if peer-evaluation is conducted seriously.

    Guidelines

    - Only grade the work that is in the student or group's
    GiT repository.

    - Double-check that the GiT repository belongs to the student
    or the group. Ensure that the work is for the relevant project
    and also check that "git clone" is used in an empty folder.

    - Check carefully that no malicious aliases was used to fool you
    and make you evaluate something other than the content of the
    official repository.

    - To avoid any surprises, carefully check that both the evaluating
    and the evaluated students have reviewed the possible scripts used
    to facilitate the grading.

    - If the evaluating student has not completed that particular
    project yet, it is mandatory for this student to read the
    entire subject prior to starting the defence.

    - Use the flags available on this scale to signal an empty repository,
    non-functioning program, a norm error, cheating etc. In these cases,
    the grading is over and the final grade is 0 (or -42 in case of
    cheating). However, with the exception of cheating, you are
    encouraged to continue to discuss your work (even if you have not
    finished it) in order to identify any issues that may have caused
    this failure and avoid repeating the same mistake in the future.

    - Remember that for the duration of the defence, no segfault,
    no other unexpected, premature, uncontrolled or unexpected
    termination of the program, else the final grade is 0. Use the
    appropriate flag.
    You should never have to edit any file except the configuration file if it exists.
    If you want to edit a file, take the time to explicit the reasons with the
    evaluated student and make sure both of you are okay with this.

    - You must also verify the absence of memory leaks. Any memory allocated on the heap must
    be properly freed before the end of execution.
    You are allowed to use any of the different tools available on the computer, such as
    leaks, valgrind, or e_fence. In case of memory leaks, tick the appropriate flag.

    Exercises

    Exercise 00 - ft_hello_garden

    Test ft_hello_garden.py (FUNCTION REQUIRED, NOT MAIN):

    • Verify the file contains a function named ft_hello_garden() (not a main program).
    • Run the function and verify it outputs exactly "Hello, Garden Community!".
    • Check that the function takes no parameters.
    • Verify the function uses print() to display the message.
    • Ensure the file is in the ex0/ directory.

    Exercise 01 - ft_plot_area

    Test ft_plot_area.py (FUNCTION REQUIRED, NOT MAIN):

    • Verify the file contains a function named ft_plot_area() (not a main program).
    • Run the function and test with different inputs like length=10, width=5 (should output "Plot area: 50").
    • Test with length=7, width=3 (should output "Plot area: 21").
    • Test with length=1, width=1 (should output "Plot area: 1").
    • Verify the function correctly calculates area using multiplication.
    • Check that the output format matches exactly "Plot area: X".
    • Ensure the file is in the ex1/ directory.

    Exercise 02 - ft_harvest_total

    Test ft_harvest_total.py (FUNCTION REQUIRED, NOT MAIN):

    • Verify the file contains a function named ft_harvest_total() (not a main program).
    • Test with inputs like day1=5, day2=8, day3=3 (should output "Total harvest: 16").
    • Test with day1=10, day2=0, day3=5 (should output "Total harvest: 15").
    • Test with all zeros: day1=0, day2=0, day3=0 (should output "Total harvest: 0").
    • Verify the function correctly adds the three values.
    • Check that the output format matches exactly "Total harvest: X".
    • Ensure the file is in the ex2/ directory.

    Exercise 03 - ft_plant_age

    Test ft_plant_age.py (FUNCTION REQUIRED, NOT MAIN):

    • Verify the file contains a function named ft_plant_age() (not a main program).
    • Test with age=75 (should output "Plant is ready to harvest!").
    • Test with age=60 (should output "Plant needs more time to grow.").
    • Test with age=61 (should output "Plant is ready to harvest!").
    • Test with age=30 (should output "Plant needs more time to grow.").
    • Verify the condition correctly uses > 60 (not >= 60).
    • Ensure the file is in the ex3/ directory.

    Exercise 04 - ft_water_reminder

    Test ft_water_reminder.py (FUNCTION REQUIRED, NOT MAIN):

    • Verify the file contains a function named ft_water_reminder() (not a main program).
    • Test with days=4 (should output "Water the plants!").
    • Test with days=2 (should output "Plants are fine").
    • Test with days=3 (should output "Water the plants!").
    • Test with days=1 (should output "Plants are fine").
    • Verify the condition correctly uses > 2 (not >= 2).
    • Ensure the file is in the ex4/ directory.

    Exercise 05 - ft_count_harvest_iterative ft_count_harvest_recursive

    Test ft_count_harvest.py (FUNCTIONS REQUIRED, NOT MAIN):

    • Verify the file contains two functions named ft_count_harvest_iterative() and ft_count_harvest_recursive() (not a main program).
    • For ft_count_harvest_iterative():
      • Test with days=5 (should output "Day 1", "Day 2", "Day 3", "Day 4", "Day 5", "Harvest time!").
      • Test with days=1 (should output "Day 1", "Harvest time!").
      • Test with days=3 and verify it counts from 1 to 3 correctly.
      • Verify the while or for loop correctly increments and stops at the right time.
      • Check that "Harvest time!" appears at the end.
    • For ft_count_harvest_recursive():
      • Test with days=5 (should output "Day 1", "Day 2", "Day 3", "Day 4", "Day 5", "Harvest time!").
      • Test with days=1 (should output "Day 1", "Harvest time!").
      • Test with days=3 and verify it counts from 1 to 3 correctly.
      • Verify the recursion correctly increments and stops at the right time (base case reached).
      • Check that "Harvest time!" appears at the end.
    • Ensure the file is in the ex5/ directory.

    Exercise 06 - ft_garden_summary

    Test ft_garden_summary.py (FUNCTION REQUIRED, NOT MAIN):

    • Verify the file contains a function named ft_garden_summary() (not a main program).
    • Test with garden_name="Community Garden", num_plants=25.
    • Test with garden_name="My Garden", num_plants=10.
    • Verify it correctly displays "Garden: [name]".
    • Verify it correctly displays "Plants: [number]".
    • Verify it displays "Status: Growing well!" exactly.
    • Check that string formatting works correctly.
    • Ensure the file is in the ex6/ directory.

    Exercise 07 - ft_seed_inventory

    Test ft_seed_inventory.py (FUNCTION REQUIRED, NOT MAIN):

    • Verify the file contains a function named ft_seed_inventory() (not a main program).
    • Check that the function uses proper type annotations: ft_seed_inventory(seed_type: str, quantity: int, unit: str) -> None.
    • Test with seed_type="tomato", quantity=15, unit="packets" (should output "Tomato seeds: 15 packets available").
    • Test with seed_type="carrot", quantity=8, unit="grams" (should output "Carrot seeds: 8 grams total").
    • Test with seed_type="lettuce", quantity=12, unit="area" (should output "Lettuce seeds: covers 12 square meters").
    • Test with an unknown unit like "unknown" (should output "Unknown unit type").
    • Verify the function correctly capitalizes the seed type in output.
    • Ensure the file is in the ex7/ directory.
