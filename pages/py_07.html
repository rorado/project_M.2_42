    PY_07

    Basics

    • Only grade the work that is in the learner's or group's
      GiT repository.

    • Double-check that the GiT repository belongs to the learner
      or the group. Ensure that the work is for the relevant activity
      and also check that "git clone" is used in an empty folder.

    • Check carefully that no malicious aliases were used to fool you
      and make you review something other than the content of the
      official repository.

    • To avoid any surprises, carefully check that both the evaluating
      and the evaluated learners have reviewed the possible scripts used
      to facilitate the grading.

    • If the evaluating learner has not completed that particular
      activity yet, it is mandatory for this learner to read the
      entire subject prior to starting the defence.

    • Remember that for the duration of the defence, no segfault,
      no other unexpected, premature, uncontrolled, or unexpected
      termination of the program, else the final grade is 0.
      You should never have to edit any file except the configuration file, if it exists.
      If you want to edit a file, take the time to explicit the reasons with the
      evaluated learner and make sure both of you are okay with this.

    • Check that only the requested files are available in the git repository.
      If not, the peer-review stops here.

    • Verify that an __init__.py file exists at the repository root (mandatory for package structure).

    • Verify that each exercise directory (ex0, ex1, ex2, ex3, ex4) contains an __init__.py file.

    Exercises

    Exercise 0 - Card Foundation

    Check that the Card class is properly implemented as an abstract base class:

    • Does Card inherit from ABC?
    • Are abstract methods defined using the abc module (not decorators)?
    • Is the play() method abstract?
    • Are get_card_info() and is_playable() concrete methods?

    Check that CreatureCard properly implements the abstract interface:

    • Does CreatureCard inherit from Card?
    • Are all abstract methods implemented?
    • Does the creature have attack and health attributes?
    • Can you create CreatureCard instances but not Card instances?

    Test the implementation with different scenarios and verify it works as expected.
    If everything works correctly, tick "Yes." Otherwise, tick "No" and the peer-review is over.

    Exercise 1 - Deck Builder

    Check that multiple card types are properly implemented:

    • Are SpellCard and ArtifactCard implemented?
    • Do they inherit from Card and implement all abstract methods?
    • Does each card type have unique behavior appropriate to its type?

    Check that the Deck class demonstrates polymorphism:

    • Can Deck store any type of Card?
    • Do deck operations (add_card, remove_card, shuffle, draw_card) work with all card types?
    • Does get_deck_stats() provide meaningful information?

    Test with different card types and verify polymorphism works correctly.
    If everything works correctly, tick "Yes." Otherwise, tick "No" and the peer-review is over.

    Exercise 2 - Ability System

    Check that abstract interfaces are properly designed:

    • Are Combatable and Magical defined as abstract base classes?
    • Do they define abstract methods appropriate to their purpose?
    • Are all methods in these interfaces abstract?

    Check that EliteCard demonstrates multiple inheritance:

    • Does EliteCard inherit from Card, Combatable, AND Magical?
    • Are ALL abstract methods from all three interfaces implemented?
    • Does the implementation make sense for a card with multiple abilities?

    Test the multiple inheritance and verify all interfaces work together.
    If everything works correctly, tick "Yes." Otherwise, tick "No" and the peer-review is over.

    Exercise 3 - Game Engine

    Check that the Strategy pattern is properly implemented:

    • Is GameStrategy defined as an abstract base class?
    • Does AggressiveStrategy implement the strategy interface?
    • Can different strategies be used interchangeably?

    Check that the Abstract Factory pattern is properly implemented:

    • Is CardFactory defined as an abstract base class?
    • Does FantasyCardFactory implement all factory methods?
    • Can the factory create different types of related cards?

    Check that GameEngine orchestrates both patterns:

    • Can GameEngine be configured with different factories and strategies?
    • Does the engine demonstrate the benefits of these patterns?

    Test pattern switching and verify flexibility is achieved.
    If everything works correctly, tick "Yes." Otherwise, tick "No" and the peer-review is over.

    Exercise 4 - Testing and Documentation

    Check that advanced interface composition is implemented:

    • Is Rankable defined as an abstract interface?
    • Does TournamentCard inherit from multiple interfaces (Card + Combatable + Rankable)?
    • Are all abstract methods from all interfaces implemented?

    Check that TournamentPlatform integrates the system:

    • Can the platform manage multiple tournament cards?
    • Do tournament operations (register_card, create_match, get_leaderboard) work?
    • Does the platform demonstrate system-level integration?

    Test the complete tournament system and verify all components work together.
    If everything works correctly, tick "Yes." Otherwise, tick "No" and the peer-review is over.

    Code Quality and Understanding

    Code Quality and Understanding

    Evaluate the learner's understanding of abstract programming patterns:

    • Can the learner explain why abstract classes are useful?
    • Can the learner explain the benefits of interfaces and multiple inheritance?
    • Does the code demonstrate proper separation of concerns?
    • Is the code well-organized and readable?

    Check for common issues:

    • Are decorators avoided (learners don't know them yet)?
    • Are all abstract methods properly implemented?
    • Is the focus on patterns rather than complex game logic?
    • Are imports between exercises handled correctly using absolute imports (e.g., from ex0.Card import Card)?
    • Are __init__.py files present in all exercise directories?
    • Are exercises executed using the module syntax (e.g., python3 -m ex0.main)?

    Discuss the design choices with the learner to assess their understanding.
    If the learner demonstrates good understanding of abstract patterns, tick "Yes."

    Code Quality & Standards

    Assess the overall code quality and standards:

    • Is the code written in Python 3.10 or higher?
    • Does the code adhere to the flake8 linter standards (no errors)?
    • Are type hints REQUIRED for all functions and methods (parameters and return values)?
    • Docstrings are NOT required for this module.
    • Is the code well-structured and readable?
    • Are variable names meaningful and descriptive?
    • Does the code follow Python best practices for abstract classes and interfaces? The code should demonstrate good programming practices and proper use of abstract patterns. Answer "Yes" if all quality standards are met, "No" if any are lacking.
